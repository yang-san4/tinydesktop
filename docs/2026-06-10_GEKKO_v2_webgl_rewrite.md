# GEKKO v2 — WebGL ネオンエンジンへの全面書き直し

2026年6月10日。GEKKO（3D FPS）を Canvas 2D ソフトウェアラスタライザ（v1、2712行）から WebGL エンジン（v2、約2100行）へ全面的に書き直した。ゲームデザイン（垂直機動・5敵種・ウェーブ制・AI テストモード）は継承しつつ、描画・ビジュアル・音・手触りを一新した。

## なぜ作り直したか

v1 のソフトウェアラスタライザは 440×330 のピクセルバッファを CPU で埋めており、描画能力が天井だった。敵はビルボードスプライト止まり、ライティングは面ごとの固定乗数、視界も `FAR_FOG=35` が限界。WebGL に移せばレトロな見た目（低解像度 + pixelated 拡大）を保ったまま、ポリゴン数・ライティング・視距離の制約がすべて1桁緩む。

## アーキテクチャ

### キャンバス3層構成

```
HTML overlay（タイトル/ポーズ/ウェーブ画面 — DOM+CSS）  z-index:10
2D HUD canvas（クロスヘア・バー・ダメージ数値・ビューモデル） z-index:5
WebGL canvas（3D シーン本体、400×300 → CSS で拡大）
```

v1 は1枚の 2D canvas に全部描いていたが、v2 は役割分担。HUD はテキスト描画が得意な Canvas 2D に任せ、3D は WebGL に専念させる。`image-rendering: pixelated` で両方を拡大するのでドット感は揃う。

### シェーダ3本

| シェーダ | 役割 |
|---|---|
| world | 頂点ライティング（太陽 + 半球環境光 + 動的ポイントライト4灯）、距離フォグ、エミッシブ |
| blend | パーティクル・影・ビーム用。ライティングなし、アルファのみ |
| sky | フルスクリーンクワッドで夕暮れグラデーション + 太陽ディスク + 星をプロシージャル生成 |

頂点フォーマットは `pos3 + normal3 + color4`（color の α をエミッシブ係数として流用）。ネオントリムや窓はエミッシブ=1 でライティングを無視させ、フォグも 55% 抜けるようにして「遠くでも光って見える」シティスケープを作った。

```glsl
vec3 col = mix(lit, aCol.rgb, em);              // エミッシブはライティング無視
vFog = clamp((dist-28.0)/40.0, 0.0, 1.0) * (1.0 - em*0.55);
```

**学習ポイント**: エミッシブ素材のフォグを弱めるだけで、夜景の「遠くの看板が滲んで光る」表現がタダで手に入る。

### ジオメトリの一元化（v1 最大の教訓）

v1 では描画用 `platforms` と装飾用 `decor` が別管理で、「見えるのに乗れない」バグを12箇所以上手で直す羽目になった。v2 では**1つの箱定義が描画と衝突の両方を生成する**:

```js
function S(cx,cy,z0,w,d,h,side,top,em){   // Solid: 見える & 乗れる
  solids.push({x:cx,y:cy,z0:z0,top:z0+h,w:w,d:d});
  pushBoxGeo(staticMesh,...);
}
function D(...){ pushBoxGeo(staticMesh,...); }  // Decor: 見えるだけ（明示的に選ぶ）
```

デフォルトが「乗れる」、装飾だけ明示的に `D()`。橋の手すりは v1 でエッジスタックの温床だったので、意図して装飾にしている。

### 静的 VBO + 動的バッファ

ワールドは起動時に1回だけ Float32Array に焼いて `STATIC_DRAW`。敵・弾・ピックアップは毎フレーム CPU でトランスフォームして `bufferSubData` で流し込む（約260箱/フレーム上限）。ブレンドパスは2回に分け、影と雲は通常アルファ、パーティクル・ビーム・グラップルワイヤーは加算合成で発光させる。

## ビジュアルの新要素

- **夕暮れシンセウェーブパレット** — 空はオレンジ→紫のグラデ + 低い太陽 + 天頂の星。フォグは紫
- **3D 敵メッシュ** — ビルボードから箱組みローポリへ。ワイバーン/ハーピーは羽ばたき（roll 回転アニメ）、ゴーレムは腕振り歩行、サーペントは体節が蛇行、ドラゴンは発光翼膜 + 尻尾3節
- **動的ライト4灯** — マズルフラッシュ・着弾・爆発が地形を一瞬照らす。`寿命×強度` でソートして上位4灯をシェーダへ
- **ブロブシャドウ** — プレイヤー・敵・ピックアップの真下に高度減衰する影。空中戦の着地点読みに効く（プラットフォーマーの定石）
- **遠景スカイライン** — 当たり判定なしの巨大タワー15本を半径40-60に配置。窓は中心向きの面だけ生成してポリゴン節約
- **アトラクトモード** — タイトル画面の裏でカメラが街を周回

## ゲームフィールの新要素

v1 から継承: jumpCD=0.15s（接地チラつき対策）、recentJet=0.4s（ジェット直後の暴発ジャンプ対策）、bestTop 着地（最高面選択）、アンチスタック（XY 押し出し時のみ 0.2u 窓でスナップ）。

v2 で追加:
- **コヨーテタイム 0.12s** — 台の縁から落ちた直後でもジャンプ受付
- **ヒットストップ 0.04s + ウェーブクリア時スローモー**
- **FOV キック** — ダッシュと高速移動で視野角が広がる
- **ヒットマーカー / キルマーカー**（白 X / 赤 X）、コンボ倍率スコア
- **RAIL（武器2）をヒットスキャン化** — 貫通レイ + 残光ビーム描画。v1 の CHARGER は弾速依存で当て感が悪かった
- **ダメージ数値を HUD 投影に変更** — v1 は 3x5 ピクセルフォントを 3D 空間に描いていたが、v2 は CPU で VP 行列投影して 2D canvas にテキスト描画。クッキリ読める

## サウンド

- SFX は全部 WebAudio 合成（発振器 + ノイズバッファ + exponentialRamp エンベロープ）
- **BGM シーケンサ新設** — 104BPM、32ステップのキック/ハット/ベース/アルペジオを先読みスケジューリング。`window._tinyDesktopSound`（デスクトップのサウンドトグル）に毎ノート従う
- ジェット噴射はバンドパスノイズのループで、噴射中だけ生成

## 検証方法（ヘッドレス Chrome + AI TEST）

node 不在の環境だったので、検証はヘッドレス Chrome で行った:

```bash
chrome --headless=new --enable-unsafe-swiftshader --enable-logging=stderr \
  --virtual-time-budget=120000 --screenshot=out.png test.html
```

テストハーネスで `requestAnimationFrame` をスタブして決定的にフレームを進め（16.7ms × 1800 = 30秒分）、ポーズ画面の **AI TEST モード**（v1 から継承）でゲーム自身に自動プレイさせた。30秒間フレームエラーゼロ、AI はウェーブ2まで進行して戦闘・グラップル・ピックアップの動作をスクリーンショットで確認。

**学習ポイント**: `--enable-logging=stderr` でページの console.error が stderr に出るので、ブラウザゲームのスモークテストは「rAF スタブ + AI 自動プレイ + スクリーンショット」で CI 的に回せる。AI テストモードを作っておくと人間がプレイしなくても回帰検証できる。

## ハマったポイント

1. **面カリングは切った** — 箱は閉じているので Z バッファだけで正しく見える。回転する敵パーツで巻き順が反転して消えるリスクを避けた（このスケールではカリングの性能メリットは誤差）
2. **合成回転行列の符号ミス** — Rz·Rx·Ry を手展開した際に r00/r02/r20 の符号を間違えた。回転行列は教科書通りに展開し直して修正
3. **初期ライティングが暗すぎた** — アルベド 0.15 × 環境光 0.4 ≒ 0.06 でほぼ黒。スクリーンショット検証で気づき、太陽 1.45 倍 + 環境光 0.6 台 + アルベド 2 倍に調整
4. **headless の `--virtual-time-budget` は rAF をあまり進めない** — タイマーは進むが描画フレームは数枚。決定的検証には rAF スタブが確実

## 追記（同日）: v1 の併存と AI モードのポーズ修正

### GEKKO V1 を比較用アプリとして復元

v1 を git 履歴（`abbe613`）から `js/fps-v1.js` として取り出し、「GEKKO V1」アプリとしてデスクトップに併存させた。v2 と DOM が衝突しないよう、ファイル内の id とクラスを一括改名している（`fps-canvas`→`fpsv1-canvas`、`window-fps`→`window-fpsv1`、注入 CSS クラス `fps-*`→`fpsv1-*`）。v1 のゲームループは自ウィンドウが最前面のときしか動かない実装なので、3本（v2 / V1 / Classic）併存しても CPU 負荷は重ならない。

**学習ポイント**: 同一ページに同系アプリを併存させるときの衝突点は「DOM id」「注入 style のクラス名」「document レベルのイベントリスナ」の3つ。前2つは機械的リネームで解決でき、リスナは元から z-index 判定（`isFpsActive`）でゲートされていたので無改修で済んだ。

### AI テストモードで ESC が効かなくなるバグ

v2 の設計バグ。ポーズは「ポインターロック解除」で発動する仕組みだが、AI モードはロックなしで動くため、AI を一度有効にすると ESC が何もしなくなり、AI を解除するためのポーズメニュー自体に戻れないデッドロックだった。

修正: 明示的な `manualPause` フラグを導入。

- AI モード中の ESC は `manualPause` をトグル（ポーズ画面が出て AI TEST ボタンに到達できる）
- 更新ゲート `canRun` と画面判定 `isPaused` の両方に `manualPause` を反映
- ポーズ画面クリック時は `manualPause` を解除し、非 AI 時のみポインターロックを再取得
- `startGame` / RESET でフラグをリセット

検証はヘッドレス Chrome の rAF スタブハーネスで6ケース（AI 中はポーズ非表示 / ESC でポーズ / 持続 / ESC 復帰 / クリック復帰 / ポーズ中 AI 解除後もポーズ維持）すべて PASS。
