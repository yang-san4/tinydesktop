## 状況
Zone1（外町 SOTOMACHI）を南側へ拡張し、探索可能な地面をほぼ2倍にした。既存の街・運河・北の門・ゾーン遷移はそのまま温存し、南に「下町(lower town)」を増築する純加算的な実装。プレイヤーの初期位置を最南端(-6,-110)へ移動し、下町→運河→大手門へ北上する動線にした。

## 経緯
### 2026-06-19
#### Zone1 を南へ増築して2倍化（方針: 拡張して作り込む／対象: 外町のみ）
ユーザー確認で「範囲を広げて作り込む（座標一律スケールではなく増築）」「外町 SOTOMACHI のみ」を選択。一律スケールは移動速度・ジャンプ距離・壁張り付き間合いが相対的に縮んで操作感が壊れるため不採用。

既存マップの境界は以下だった:
- 南岸 `S(0,-27,-1,68,58,1)` → x[-34,34], y[-56,2]
- 運河 y[2,9] / 北岸 y[9,32] / 北壁・門 y=29（exitGate y:31 で Zone2 へ）

南岸の南端 y=-56 に接して新スラブを増設:

```js
// SOUTH EXTENSION: 下町（探索地面を2倍化）
S(0,-86,-1,80,60,1,C_DIRT,C_DIRT);   // x[-40,40], y[-116,-56] = 4800u² 追加
D(-6,-57,0.02,3.6,118,0.02,...);      // 主街路(x=-6)を南端まで延伸
D(14,-86,...);D(0,-86,...);D(0,-108,...); // 東路+市場/南の十字路
```

増築内容: 町家2列＋遠景列、裏路地の屋台市場（第2の壁張り付き角）、蔵+櫓、南東に鳥居+社、南端の城壁(rampart)+四隅櫓、提灯、桜、地蔵（チェックポイント兼用）、かぎ縄アンカー＋屋根上小判、蛍。敵はashigaru巡回6+櫓上のarcher2を追加。

```js
checkpoint=[-6,-110,0];
player.x=-6;player.y=-110;player.z=0.5;   // 最南端スタート、北上動線
hintTrigs.push({y:-104,msg:'SOTOMACHI — the lower town...'});
```

**学習ポイント**:
- 落下リスポーンは `player.z<VOID_Z`（高さ）+checkpointベースで、xy境界をハードコードしていない。地面を solids ベースの `groundTopAt` が拾うため、`S()` で地面スラブを足すだけで歩行可能域を拡張できる＝**マップ拡張がほぼ純加算で済む設計**になっていた。
- hintTrig は `player.y > hint.y` で発火。南(小さいy)スタート→北進(y増加)なので、南端のヒントが先・北のヒントが後、と自然に順序発火する。
- スケール一律2倍は実装が速いが「世界に対する自機性能」が変わり操作感が劣化する。**「広さ」を増やす要求は増築（新エリア+配置）で応える**のが正解。
- 既存スラブ(幅68)に対し新スラブを幅80にしても、接合は南端 y=-56 のみ・z 同一平面なので段差は出ず「町が広がる」自然な見えになる。

#### 外周の落下対策: 土塁＋杉木立で囲う
拡張後、外周（特に東西）に壁がなく端から落ちて奈落になるのが指摘点だった。「壁 or 森/海」の要望に対し、ニンジャ夜戦の雰囲気に合う**土塁(dorui)＋暗い杉森**で対応。

ジャンプ性能を確認して土塁高さを決定:

```js
var JUMP_VEL=6.8, GRAVITY=18;   // 単発到達高 ≈ v²/2g = 6.8²/36 ≈ 1.28u
// 二段ジャンプでも実効 ~2.5u 程度 → 土塁 h=4.5u なら平地から越えられない
```

ヘルパを2つ追加:

```js
function sugi(x,y,h,deco){       // 暗い杉(針葉樹): テーパした緑箱3段。deco=true で当たり判定なし
  var th=0.30,g0=[0.07,0.16,0.10],g1=[0.11,0.22,0.14];
  if(deco)D(x,y,0,th,th,h,C_WOOD,C_WOOD);else S(x,y,0,th,th,h,C_WOOD,C_WOOD);
  D(x,y,h*0.38,2.0,2.0,h*0.32,g0,g1,0); ...
}
function dorui(cx,cy,z0,w,d,h){  // 土塁: 土の盛り土 + 苔のキャップ
  S(cx,cy,z0,w,d,h,C_DIRT,[0.14,0.22,0.13]);
  D(cx,cy,z0+h,w+0.5,d+0.5,0.25,[0.12,0.19,0.11],[0.16,0.26,0.16]);
}
```

配置: 東西土塁を2段（原幅域 y[-57,33] は x=±35.5、下町の広幅域 y[-117,-55] は x=±41.5）で歩行端のすぐ外に置き、y方向にオーバーラップさせて継ぎ目の隙間をなくした。土塁の外側に杉森をループで撒く（`deco=true`＝当たり判定なしのバックドロップ）。森だけ当たり判定を持たせると solids が膨らむので、**衝突は土塁が担当・森は純描画**に役割分担。町内にも数本だけ実体杉(trunk=S)を「隠れる遮蔽物」として置いた。

**学習ポイント**:
- 境界の作り方は「越えられない高さの実体(土塁)＋雰囲気を出す装飾(森)」に分離すると、当たり判定コスト（solids 数）を最小にしつつ見栄えを盛れる。森を全部コライダーにするのは無駄。
- 壁高は感覚で決めず、`JUMP_VEL²/2·GRAVITY` で到達高を計算して「越えられない最小高」に合わせる。むやみに高くすると圧迫感が出るし、低いと脱出される。
- 拡張で幅が段違い(68→80)になった境界は、土塁を2セグメントに分け y でオーバーラップさせると角の隙間（到達不能な void ポケット）を塞げる。

#### 操作説明と実バインドのズレ修正（壁張り付き: C → SHIFT）
ゲーム中の操作説明が実操作と合っていないとの指摘。調べると壁張り付きの実バインドは元々 `C` キーだったが現状は `SHIFT`（`onShiftPress`: 壁が近ければ cling、なければしゃがみ）に変わっており、**古い `C` 表記が2か所だけ残存**していた。

- `_pcControls()`（ポーズ/タイトルの操作パネル）は既に全て一致（SHIFT=SNEAK/CLING も正しい）。実バインド: 左クリック=斬撃 / 右クリック・F=手裏剣 / SPACE=ジャンプ / SHIFT=忍び歩き・壁張り付き / 方向2回=回避 / E=かぎ縄照準 / ESC=ポーズ。
- 直した残存箇所:
  - `hud.fillText('C: WALL CLING',...)`（壁の近くで出る画面下のアフォーダンス表示）→ `'SHIFT: WALL CLING'`
  - `hintTrigs.push({y:-31,msg:'C: WALL CLING …'})`（チュートリアルヒント）→ `'SHIFT: …'`
- 他のキー参照テキスト（`press SPACE`=二段跳び / `press E`=かぎ縄 / `(SHIFT)`=鳴く床 / `SHIFT: SNEAK …`）は全て実バインドと一致を確認。

**学習ポイント**: キーバインドを変更したら、入力ハンドラ(`switch(e.code)`)だけでなく **(1)操作パネル文言 (2)HUDアフォーダンス (3)チュートリアル/ヒント文 (4)スクロール取得時の説明** の4系統すべてを grep で洗うこと。今回は (2)(3) に旧 `C` が取り残されていた。`C` キーは現状ハンドラに case が無く完全な dead key だったため、表示だけが嘘をついていた。
