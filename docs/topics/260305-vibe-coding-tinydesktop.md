# バイブコーディングで「小さなデスクトップOS」を作ってみた

> **注**: この記事自体も AI（Claude）が執筆しています。プロジェクトのコードも、この振り返り記事も、人間の指示と判断のもとで Claude が書きました。

## はじめに

「バイブコーディング（Vibe Coding）」という言葉を最近よく聞く。AIにざっくりした指示を出して、対話しながらコードを書いてもらうやり方だ。

実際どこまでいけるのか試してみたくて、**ブラウザの中で動くミニチュアPC**を作ってみた。名前は **TinyDesktop**。CRTモニターの中に小さなOSが動いていて、ウィンドウを開いたり、ゲームで遊んだり、テーマを切り替えたりできる。

結果から言うと、**7日間・約20,000行**のコードが出来上がった。フレームワークなし、ビルドツールなし。HTML / CSS / バニラJSだけ。

https://yang-san4.github.io/tinydesktop/

---

## 何ができるか

440×330ピクセルの小さな画面の中に、15個のアプリが入っている。

**ツール系**
- 時計（アナログ+デジタル）
- カレンダー
- ターミナル（30以上のコマンド対応）
- ピアノ（キーボード演奏可能）
- ブラウザ（TinyWeb — 架空サイトを巡回できる）
- タイマー（砂時計ウィジェット）
- ダビデ像（デスクトップ彫刻）

**ゲーム系**
- GEKKO（3D FPSアリーナ）
- GEKKO Classic（レトロ版）
- テトリス
- ソリティア
- ピンボール（3Dレール）
- マインスイーパー
- 3Dメイズ

それに加えて、5つのテーマ切替、フォルダシステム、右クリックメニュー、デスクトップペット、ドラッグ&リサイズ、タスクバー、モバイル対応まである。

## 開発のタイムライン

### Day 1（2/18）— 骨格を一気に
初日で基本的なデスクトップの形ができた。ウィンドウマネージャー、タスクバー、時計、カレンダー、ターミナル、アクアリウム、CRTモニターの外観。さらにテーマシステム、ブラウザ、フォルダ、FPSゲームの原型まで。

初日だけで5コミット。AIに「こういうのが欲しい」と伝えると、丸ごとコードが返ってくる。自分はそれを眺めて、「もうちょっとこう」と調整する。

### Day 2-3（2/19-20）— モバイル対応の泥沼
ここが一番コミット数が多い（11回）。モバイルのタッチ操作、特にFPSゲームのツインスティック操作を何度も調整した。

バイブコーディングの弱点がここで見えた。**「触った感触」はAIに伝えにくい**。「もうちょっと感度を上げて」「ボタンが小さい」みたいなフィードバックの繰り返し。数値の微調整は結局、自分の感覚に合うまで何度もやり直した。

一方でGEKKOのゲーム性改善（敵AI、コリジョン、ジェットパック物理）はAIが得意な領域だった。「敵が壁に引っかかる」→ 原因特定 → 修正の流れがスムーズ。

### Day 4（2/25）— アプリ追加ラッシュ
テトリス、ソリティア、ターミナルの大幅強化を一日で。**「テトリスを追加して」と言うだけで、回転テーブル・ウォールキック・ゴーストピース付きの完全なテトリスが出てくる**。ソリティアも同様。Canvas描画でピクセルアートのカードが並ぶ。

ここでセキュリティ強化も実施。ターミナルにsudo/su/rm -rfなどの危険コマンドのハンドリングを追加。

### Day 5-6（3/2-3/3）— ピンボール
ピンボールは一番手間がかかった。最初のバージョンから3回作り直している。物理演算が入ると、「なんか違う」の感覚を言語化するのが難しくなる。「ボールの跳ね返りが硬い」「フリッパーに吸い付く感じがない」——こういう曖昧なフィードバックでも、AIは何かしら返してくる。でも正解に辿り着くまでに試行回数が必要だった。

### Day 7（3/4-5）— 仕上げ
アイコンのテーマ対応、自動グリッド配置、Gameフォルダへの整理。地味だけど、こういう「使い勝手」の改善はAIとの相性が良い。「macOSみたいにアイコンを左上から並べたい」で通じる。

## バイブコーディングで分かったこと

### AIが得意だったこと

**1. ゼロからの雛形生成**
「CRTモニター風のデスクトップを作って」で、HTML構造、CSSのCRT効果（スキャンライン、ビネット、グロー）、ウィンドウマネージャーのドラッグロジックまで一発で出てくる。自分でゼロから書いたら丸一日かかる部分が、数分で形になる。

**2. 既知のゲームの実装**
テトリス、ソリティア、マインスイーパー。ルールが明確なゲームは「作って」で完成品が出る。しかもピクセルアート風のCanvas描画まで含めて。

**3. バグの原因特定**
「敵が壁を突き抜ける」「アイコンがテーマ切替で色が変わらない」——症状を伝えれば、コードを読んで原因を見つけてくれる。自分で数百行のコードを目grepするよりずっと速い。

**4. リファクタリングと設計改善**
「ハードコードされた色をCSS変数に変えて」のような構造的な改善は、AIが全体を見渡して一括で書き換えてくれるので効率的。

### AIが苦手だったこと

**1. 「感触」の調整**
モバイルのタッチ感度、ピンボールの物理、FPSの操作感。これらは数値を微調整しながら「触って→直して」を繰り返す必要がある。AIにはその「触ってみた感覚」が分からないから、フィードバックの言語化が自分の仕事になる。ここが一番疲れた。

**2. 視覚的な完成度の判断**
CRTのグロー効果の強さ、アイコンの色バランス、ウィンドウの影の濃さ。「いい感じ」かどうかは画面を見ないと分からない。AIが生成したものをそのまま使えることは少なくて、毎回「もうちょっと暗く」「影を柔らかく」みたいな修正が入る。

**3. 大きなファイルの全体把握**
GEKKOのメインファイル（fps.js）は2,700行ある。AIは一度に全体を見渡せるわけではないので、「この変数ってどこで使ってる？」みたいな確認を先にやらないと、別の場所を壊す修正を入れてくることがある。

### 人間の役割

バイブコーディングで人間がやることは、**設計判断**と**品質の最終判定**だった。

- 「次にどのアプリを追加するか」はAIに聞いても答えが出ない
- 「テーマを5つ用意する」と決めるのは自分
- 「このピンボールは面白くない、作り直そう」と判断するのも自分
- AIは「どう作るか」は得意だが「何を作るか」「これでいいか」は人間が決める

## 技術的に面白かったところ

### フレームワークなしの設計判断

React も Vue も使わなかった。440×330px の小さな画面に15個のアプリが詰まっているだけで、SPA ルーティングも状態管理ライブラリも不要だった。

各アプリは独立した IIFE（即時実行関数式）で、グローバルを汚さない。アプリ間の通信は `window._tinyDesktop` のような最小限の公開APIだけ。これで十分だった。

```
js/main.js      — ウィンドウマネージャー（589行）
js/fps.js       — GEKKO本体（2,712行）
js/pinball.js   — ピンボール（2,302行）
js/browser.js   — TinyWeb（2,175行）
js/terminal.js  — ターミナル（1,411行）
...
```

### CSS変数によるテーマシステム

5テーマ（Vapor / Wafuu / Mokume / Classic / Aqua）の切替は、`:root` の CSS 変数を差し替えるだけ。新しいアイコンを追加するときも `var(--accent2)` のような意味的な変数を使えば、テーマ対応が自動的に済む。

```css
/* テーマ定義 */
#screen.theme-osx {
  --accent: #3478f6;
  --accent2: #1a5ac7;
  --surface: #d0d8e4;
  --body-deep: #f0f4f8;
}

/* アイコンはテーマを意識しない */
.di-fps {
  background: var(--surface2);
  border-color: var(--accent2);
}
```

### Canvas ベースのゲーム群

全ゲームが `<canvas>` で描画されている。フレームワークやゲームエンジンは使わず、`requestAnimationFrame` + 手書きの描画ループ。ピクセルアート風の描画は `image-rendering: pixelated` で拡大時もドットが潰れない。

## まとめ

バイブコーディングは「10倍速い」とかいう話ではなかった。**AIが一瞬で出すコードの品質を、自分の目と手で検証する時間は普通にかかる**。

でも、一人で7日でこの規模のものを完成させられたのは、間違いなくAIのおかげだ。テトリスのゲームロジックを自分で書かなくていい。CSSグラデーションの微調整に時間を使える。「作業」を減らして「判断」に集中できるのがバイブコーディングの本質だったと思う。

一つ確実に言えるのは、**AIに任せきりにはできない**ということ。ピンボールを3回作り直したように、最終的な品質は人間の「これでいいか？」という問いかけにかかっている。AIは驚くほど速く「それっぽいもの」を作るが、「本当にいいもの」にするのは自分自身の仕事だった。

コードは全てGitHubで公開している。興味があれば触ってみてほしい。
