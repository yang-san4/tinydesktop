# KAGE — 和風忍者アクションRPG 新規開発（マルチエージェント編成）

2026年6月11日。tinydesktop に8本目のゲーム「KAGE」を追加した。三人称視点の和風忍者アクションRPG。月夜の城下町に潜入し、大手門を突破し、天守を登って殿様を闇に葬る、1周約10〜15分の攻略型。js/kage.js 約2,900行。

## 開発体制（Workflow によるオーケストレーション）

ユーザー要望によりマルチエージェント編成で開発。dfplus.io 専用の orchestrator エージェントは仕事用のため使わず、Workflow ツールで代替した。

1. **設計フェーズ**: 戦闘デザイナー / レベルデザイナー / アート&オーディオの3視点エージェントを並列起動 → ディレクターエージェントが統合。矛盾解消（鉤縄射程14u vs 8u → 8u 採用等）と削減順序（畳返し→桜枚数→屋根表現）まで含む31項目の実装仕様が出力された
2. **実装フェーズ**: 統合仕様に基づき本体（私）が実装。GEKKO v2 エンジン（箱ジオメトリ一元化・頂点ライティング・ヘッドレス検証）を三人称化
3. **レビューフェーズ**: ゲームロジック / WebGL / tinydesktop 統合の3視点エージェントが並列レビュー → 14件の指摘（重複除き9件）を全て修正

## ゲーム内容

- **3ゾーン攻略**: 城下町（JOKAMACHI）→ 大手門・堀（OTEMON）→ 本丸・天守（TENSHU）。ゲート通過でフェード暗転 → 次ゾーンを再構築
- **ステルス**: 敵は3段階警戒（平常→疑念?→警戒!）。視覚は90°コーン+遮蔽判定、聴覚は歩行6u/忍び歩き(Shift)1.5u/ダッシュ9u の3D距離。非警戒の敵は背後から一撃の闇討ち（墨色の暗転+白い斬線）。手裏剣で提灯を消すと周囲の敵の視界が狭まる
- **剣戟**: 3段コンボ（3段目は回転斬り+踏み込み）、先行入力バッファ、ヒットストップ、背面ボーナスダメージ
- **移動キット**: ジャンプ・段差はステップアップで歩いて登れる・Qダッシュ（無敵）・巻物①二段跳び・巻物②鉤縄（金色アンカーへ射出）
- **RPG-lite**: 心拍3のHP、小判ドロップ、おにぎり回復、地蔵=チェックポイント（死亡で小判半減）、無警戒クリアで称号「影」+小判50
- **ボス「殿様」**: 2フェーズ。居合斬り（鞘が赤く発光する1秒の構え→120°扇形）と扇子3連投。HP半分で羽織を脱ぎ目が赤く点灯、高速化。背面取りで大ダメージ＝ステルスの文法をボス戦で回収
- **演出**: 月と星の夜空シェーダ、提灯の揺らぐ暖色ライト（最寄り4灯選抜）、桜吹雪、地を這う霧、なびく赤マフラー、ダッシュ残像
- **音**: 都節ペンタトニック（D-Eb-G-A-Bb）の2レイヤーBGM——潜入84BPM（疎な太鼓+琴の下行）/警戒132BPM（締太鼓+琴トレモロ）。SFX は琴・太鼓・拍子木・刃音を WebAudio 合成

## レビューで潰した主な不具合

- AI TEST のウェイポイントがゾーン遷移でリセットされず堀に直行して死亡ループ（通しプレイの失敗原因をレビューが正確に特定）
- グラップル中に死亡すると復活後にステイルなアンカーへ射出される
- ボス初撃で警戒カウントが増え、称号「影」が原理的に達成不可能
- 壁密着時にカメラが壁内部へめり込む（cameraClip の最小距離フロア撤廃）
- カメラの床スナップが頭上の屋根を拾って屋根上へワープ
- 門の扉幅が開口より狭く、脇をすり抜けてギミックを丸ごとスキップできた
- 敵の壁すり抜け追跡・弓兵の城壁からの自滅落下・堀上空での空中静止
- 8u 上の階の敵が足音で警戒し続ける（聴覚・警戒維持の3D距離化）

## 検証

ヘッドレス Chrome + rAF スタブ + AI TEST モードで、**AI がタイトルから殿様撃破まで自力で完走**することを確認（巻物2本回収・レバー・門番撃破・東面階段からの天守登攀・ボス戦込み、約90秒）。スモーク7ケースも全 PASS。取り損ねによる詰み対策として巻物のマグネット吸着と御殿の予備鉤縄も追加した。

## 既知の制限

- モバイルは非対応（設計判断）。タイトルに「PLAY ON DESKTOP — PC ONLY」を表示
- 設計仕様のうち畳返し（ボス第3攻撃）・敵への鉤縄スタン・提灯持ちの個別差分の一部はスコープ削減（設計フェーズの削減順序に従った）

## 追記（同日）: 鉤縄の信頼性修正

ユーザー報告「鉤縄が使えないかも」。ハーネスで人間操作を忠実に再現（ポインターロック模擬 + カメラ操作フックを `_kageTest` に追加）して調査した結果、発射と引き寄せは機能しており、**到達後にまともに着地できない**のが実態だった。原因は4つの複合:

1. **解除時のモメンタム過多** — 到達解除時に引き寄せ速度がそのまま残り、屋根を飛び越して反対側へ落ちる
2. **アンカーの設置位置** — 櫓のアンカーが庇より低い位置にあり、引き寄せ先が屋根の下だった
3. **壁接触で引き寄せ即キャンセル** — 壁面に沿って吊り上がる挙動が成立しなかった
4. **庇の側面が横速度をゼロ化** — 跳ね上がり中にスラブ側面へ触れると、縁の 0.24u 手前で垂直往復して落ちる

修正: 解除を**スマートリリース**化（アンカー直下の足場の高さを `groundTopAt` で読み、確実に乗り越えられる初速を逆算して跳ね上げ + 足場方向へナッジ）、解除後 0.8 秒の**マントリング補助**（横方向の押し込みを毎フレーム再適用）、壁接触キャンセルを撤廃して**スタック監視**（0.4秒進捗なしで自動解除）に置換、照準判定を3D化（ほぼ真上のアンカーは視線不問）、アンカー5箇所を着地面の真上へ移設。

検証: 門の屋根 / 櫓の上 / 城壁の歩廊の3ケースで「発射→引き寄せ→着地」の z 軌跡を確認、全 PASS。AI 完全通し（鉤縄2回使用込み）も継続して勝利。
